### My Achievements
- **Identified that `waltid.ts` was a leaky abstraction masking the per-company wallet design:** The monolith called `listWalletCredentials` without a wallet context, silently returning operator wallet data — a subtle correctness bug that would have been invisible until a company-specific credential check was needed. Spotted and fixed it as part of the cleanup rather than leaving it for a later bug report.
- **Designed the `walt-generic-client.ts` extraction boundary correctly:** Rather than just moving all of `waltid.ts`, identified which functions were truly "generic" (credential issuance, VP verification) vs wallet-context-specific (`listWalletCredentials` already in `wallet-api-client.ts`) and split accordingly — avoiding a renamed version of the same problem.
- **Closed the ADR lifecycle properly:** ADR-002 had been in `Proposed` status even though the architecture it described was fully implemented. Marking it `Accepted` with the completion date makes the decision record accurate as an audit trail — important for Gaia-X compliance documentation.

### Key Learnings
- **Monolith service files (like `waltid.ts`) tend to accumulate misrouted concerns:** When a file provides "utility" functions, it becomes a dumping ground for logic that doesn't yet have a proper home. The right approach is to extract into bounded files (`company-wallet-service.ts`, `wallet-api-client.ts`, `walt-generic-client.ts`) where the responsibility is clear from the name.
- **Credential endpoint data ownership must be enforced at every layer:** The original `listWalletCredentials()` call with no wallet context was architecturally inconsistent with the per-company wallet design decided in ADR-002. This shows that architectural decisions need to be propagated through ALL endpoints, not just the primary ones — secondary endpoints can be overlooked during a large refactor.
- **ADR status is part of the documentation value:** An ADR in `Proposed` state on a shipped feature gives readers the wrong impression about design certainty. Tracking the ADR lifecycle (Proposed → Accepted → Superseded) costs nothing but significantly improves architectural clarity for future team members and auditors.

### Key Learnings
- **Claude's brainstorming skill is a structured-interview tool, not a code-generation tool:** The skill deliberately forces one multiple-choice question at a time before writing anything. That cadence feels slow for an expert who already knows the answer, but it surfaces trade-offs that get glossed over in a direct "just write the design" prompt — I caught at least two hidden assumptions I would have missed (HTTP-01 wildcard impossibility; ArgoCD Image Updater needing write-scope SSH key).
- **Cross-session continuity is a first-class design concern when working with Claude Code on long tasks:** The combination of (a) a markdown spec checked into the repo and (b) a structured memory entry in `.claude/projects/*/memory/` is the right pattern for surviving context-window or account switches. Keeping the spec file authoritative and using memory as a *pointer + decision summary* (not a replacement for the spec) preserves both auditability and resumability.
- **The `/caveman` mode is genuinely useful for interview-style flows:** Dropped token usage noticeably during the 22-question exchange while preserving technical exactness. Good fit when the operator is already expert and just needs to pick options — less good when the operator needs the reasoning.
- **Writing a spec before a plan is the right break point:** The design doc commits the *what* and *why* without locking in the *how*. When the user reviews the spec and approves, the subsequent `writing-plans` invocation has an anchor — if the plan drifts, it's visible as a spec-vs-plan diff rather than Claude quietly deciding to do something different.

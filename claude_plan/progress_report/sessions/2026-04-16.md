### What Was Done
Debugged and resolved three sequential Gaia-X compliance failures during company onboarding — each requiring root-cause analysis before the next error became visible. First failure: VC issuance returned 400 from walt.id issuer API because (a) `credentialConfigurationId` was `UniversityDegree_jwt_vc_json` which doesn't exist in `credential-issuer-metadata.conf`, and (b) a public JWK was being passed instead of a private key. Fix: bypass the issuer API entirely — export the RSA private key from the wallet transiently (`loadPrivateKey=true`), convert JWK → PEM via `crypto.createPrivateKey`, sign the VC JWT locally using `jsonwebtoken` with `iss` in the JWT header. Second failure: Gaia-X compliance returned `"DID does not contain x5u nor x5c"` — the DID document's RSA verification method had no certificate. Fix: generate a self-signed X.509 cert during provisioning (ASN.1 DER built from scratch using the deleted `VPSigner` code as reference), store PEM in DB (`rsaCertPem`), expose base64-DER in DID document `publicKeyJwk.x5c`, and include in JWT headers. Third failure: Prisma migration failed with `relation "Company" does not exist` — the SQL used quoted uppercase `"Company"` but the actual PostgreSQL table is `companies` (plural, lowercase). Fix: corrected to `ALTER TABLE "companies" ADD COLUMN IF NOT EXISTS "rsaCertPem" TEXT` and resolved the failed migration via `prisma migrate resolve --rolled-back`. Fourth fix: x5c values in DID doc and JWT headers were being set to full PEM strings (with `-----BEGIN CERTIFICATE-----` markers) — RFC 7515 §4.1.6 requires raw base64-encoded DER with no markers. Added `certPemToX5cValue()` utility and applied it at all x5c embedding sites.

- **Caught the RFC 7515 x5c format violation before production testing:** Identified that embedding full PEM (with `-----BEGIN CERTIFICATE-----` markers) in x5c is a standards violation that would cause a 500 at the compliance endpoint — and fixed it proactively rather than discovering it at runtime.

### Key Learnings
- **Debugging layered SSI failures requires isolating one error at a time:** Each fix revealed the next error — VC 400 → missing x5c → wrong table name → wrong x5c format. Trying to fix all at once would have been impossible. The discipline of "solve one root cause, then restart" is essential for complex cryptographic pipelines.
- **PostgreSQL table names in migrations must be verified against the actual DB, not inferred from the Prisma model name:** Prisma maps `Company` model → `companies` table (lowercase, plural) by default, but this isn't obvious. Running `psql \dt` to check before writing SQL prevents migration failures in production.
- **JWT x5c is raw base64 DER, not PEM — and this distinction matters at standards-compliant endpoints:** RFC 7515 §4.1.6 is explicit: x5c must be base64-encoded DER with no PEM headers. The Gaia-X compliance endpoint enforces this strictly. Always strip PEM markers before embedding in JWT or JWK headers.

### What Was Done
Continued GXDCH compliance debugging after the morning session (continued in Cursor after hitting Claude Code context limit) — focused on VP-JWT format failures. Worked through a chain of four progressive errors: (1) "stream is not readable" HTTP 500 — GXDCH was not reading the request body at all; fixed by sending raw JWT via `--data-binary` instead of JSON body; (2) "Not a Valid JWT and Was Not Decoded" HTTP 400 — VP-JWT format malformed; decoded `/tmp/vp_jwt_debug.txt` and `/tmp/vc0_debug.txt` to inspect headers/payload; found non-standard `typ: vc+ld+json+jwt` and `vp+ld+json+jwt`, VP payload missing `iss`/`sub` JWT registered claims (using JSON-LD `issuer`/`holder` fields instead), and no `vp` claim wrapper around the JSON-LD; (3) Fixed VC JWT endpoint `Content-Type` from `application/vc+jwt` to `application/vc+ld+json+jwt` to match the actual `typ` in the signed JWTs; (4) Confirmed from GXDCH 2.11.0 Swagger docs that `iss` must appear in the JOSE header of the VP JWT — non-standard per RFC 7519 but explicitly required by GXDCH. Additionally confirmed: the VP's three embedded VCs (LegalPerson, LeiCode, TermsAndConditions) are structurally correct with proper LEI code; the stored LP VC JWT served at the `vcid` URL also lacks `iss` in its JOSE header — so GXDCH would reject it independently even if VP format were fixed; and both VP and VC JWTs contain `x5c` in the JOSE header which breaks OpenSSL 3.x in the compliance service. Session ended with all root causes identified and documented in CLAUDE.md gotchas; no code fixes beyond the Content-Type change were applied.

### Key Learnings

- **GXDCH requires `iss` in JOSE header for both VP JWT AND each individual VC JWT it dereferences:** This non-standard requirement (RFC 7519 defines `iss` as a payload claim) applies to both the submission and the `vcid`-resolved credential. Missing it in either one fails the check.

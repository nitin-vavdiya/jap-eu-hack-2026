### Key Learnings

### What Was Done
Completed a full auth hardening pass on `portal-tata-admin` (port 3002). The session started with a diagnostic discussion: `GET /api/companies/me` was returning 404 because `AUTH_ENABLED=false` was injecting a mock `sub: 'mock-company'` that had no `CompanyUser` row in the database. Rather than applying a fallback fix, the user correctly identified this as an anti-pattern and directed removal of the entire `AUTH_ENABLED` bypass. The `getMockUser()` function and all three conditional bypasses were stripped from `backend/src/middleware/auth.ts` — Keycloak JWT validation is now unconditional. The `/me` endpoint was tightened from bare `authenticate` to `requireRole('company_admin')`, ensuring only company admins can call it. On the frontend, 5 independent calls to `/api/companies/me` (in `App.tsx` ×2, `CarList`, `CreateCar`, `CompanyProfile`) were consolidated into a single `CompanyProvider` context (`src/context/CompanyContext.tsx`) that fetches once at app startup; all pages now consume the result via `useCompany()`. A new `CompanyProfile` page was built at `/company` with 5 tabs: Overview, Credentials, Gaia-X & Compliance, EDC Infrastructure, and DID Document — fulfilling the requirement for a menu showing onboarding-generated certificates and Gaia-X data. `CLAUDE.md` was updated with explicit rules on the always-on auth pattern, the context-based company data pattern, and named anti-patterns to avoid.

### Key Learnings
- **`AUTH_ENABLED` toggles are always an anti-pattern:** A bypass that injects a mock user `sub` that doesn't exist in the database creates a different failure mode than real auth — the endpoint silently returns wrong data or 404, masking auth integration bugs until production. Always-on auth with Keycloak running locally is the correct posture, even in development.
- **Data ownership must be server-enforced, not client-trusted:** The right model is: token → `req.user.sub` → `CompanyUser.keycloakId` lookup → return only that user's data. The frontend never passes `companyId` for writes — the backend derives it. Any endpoint that accepts a client-supplied `companyId` for a mutation is an IDOR vulnerability by design.
- **React context eliminates redundant API fan-out:** When multiple components in the same app need the same identity data, a single context provider is the correct pattern — not per-page `useEffect` fetches. The `CompanyProvider` + `useCompany()` pattern is now the standard for any cross-cutting data in this portal, and the rule is documented in CLAUDE.md so it survives future Claude sessions.
